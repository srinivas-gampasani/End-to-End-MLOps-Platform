## Overview

A complete **MLOps platform** covering the full ML model lifecycle: experiment tracking and versioning (MLflow), statistical drift detection (PSI + Kolmogorov-Smirnov), Kubernetes-native canary deployments with automated promote/rollback decisions, and Prometheus/Grafana observability — all wired together and demonstrated end-to-end with a real trained churn-prediction model.

```
├───────────────┴───────────────┴───────────────┴─────────────┤
│                     FastAPI Control Plane                    │
└─────────────────────────────┬─────────────────────────────────┘
                               │
              ┌────────────────┼────────────────┐
              ▼                ▼                ▼
        Train Champion   Detect Drift     Deploy Canary
        + Challenger     vs Reference     (10% → 100%)
              │                │                │
              ▼                ▼                ▼
        MLflow Registry   Prometheus       Promote / Rollback
        (versioned)       Alerts           (automated decision)
```

### CI/CD Pipeline (GitHub Actions — `ci/mlops_pipeline.yml`)

```
test → train_challenger → validate_against_champion → drift_check
  → deploy_canary (10% traffic, K8s) → monitor 30min → promote_or_rollback → Slack notify
```

### Kubernetes Manifests (`k8s/`)

- `deployment-champion.yaml` — production deployment + service + health probes
- `deployment-canary.yaml` — canary deployment + Istio VirtualService traffic split
- `namespace-and-scaling.yaml` — HPA (CPU/memory autoscaling) + scheduled drift-check CronJob

## Drift Detection Methodology

- **PSI (Population Stability Index)**: bins reference distribution into deciles, compares current distribution against those bins. PSI < 0.1 = stable, 0.1–0.2 = moderate, > 0.2 = significant drift requiring investigation.
- **Kolmogorov-Smirnov test**: two-sample test for distribution equality; p-value < 0.05 flags statistically significant shift.
- **Data quality checks**: null rate thresholds, infinite value detection, constant-column (pipeline bug) detection.

## Canary Decision Logic

1. **Hold** if canary has insufficient traffic (< 30 requests) — keep collecting data
2. **Rollback** if canary success rate < 95% threshold, or latency > 1.5× champion
3. **Promote** if canary success rate ≥ champion success rate (within 1% tolerance)
4. Otherwise **hold** and continue monitoring
